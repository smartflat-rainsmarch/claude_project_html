# 회의실 예약 시스템 구현 계획서

> 작성일: 2026-03-20
> 참고: docs/reservation.md (기획서), docs/reservation_design.md (설계서)

---

## 구현 Phase

### Phase 1: View 화면 (type_reservation_view.html)

| # | 작업 | 설명 |
|---|------|------|
| 1-1 | HTML 구조 | 상태 영역, 상세 정보, 다음 예약, 오늘 전체 일정 |
| 1-2 | CSS 스타일 | 큰 폰트(60~80px 상태), 색상별 상태, 카드형 레이아웃 |
| 1-3 | 상태 판단 로직 | 현재시간 기준: 회의중/사용가능/곧시작/예약없음 |
| 1-4 | 지난 회의 흐리게 | opacity 0.3 + 취소선, 현재 회의 강조 |
| 1-5 | 자동 갱신 | 30초 시간 업데이트, 1분 데이터 갱신 |
| 1-6 | 숨김 필드 처리 | is_hide_* 값에 따라 영역 비노출 |
| 1-7 | URL 파라미터 | room_no, hm_idx 읽기 |

**완료 기준:** 회의실 문 옆에서 현재 상태가 크고 예쁘게 보임

---

### Phase 2: Upload 화면 (type_reservation_upload.html)

| # | 작업 | 설명 |
|---|------|------|
| 2-1 | HTML 구조 | 회의실 선택, 날짜 선택, 입력 폼, 예약 목록 |
| 2-2 | 입력 폼 | 팀명, 예약자(숨김), 인원(숨김), 시간, 제목(숨김), 부제(숨김) |
| 2-3 | CRUD | 등록, 수정, 삭제, 초기화 |
| 2-4 | 유효성 검사 | 필수값, 시간 순서, 중복 검사 |
| 2-5 | 목록 표시 | 선택된 회의실/날짜 기준, 클릭→폼 로드 |
| 2-6 | 저장 API | hm_other JSON → PUT /homes/{hm_idx} |

**완료 기준:** 예약 등록/수정/삭제 동작, 중복 방지

---

### Phase 3: 데이터 연동 + 7일 cleanup

| # | 작업 | 설명 |
|---|------|------|
| 3-1 | hm_other 읽기/쓰기 | 기존 home.php API 활용 |
| 3-2 | 7일 지난 데이터 삭제 | 저장 시 7일 이전 end_datetime 항목 자동 제거 |
| 3-3 | View ↔ Upload 연동 | 같은 hm_other 데이터 공유 |
| 3-4 | 에러 처리 | API 실패, 네트워크 에러 |

**완료 기준:** E2E 동작 — 예약 등록 → View에서 즉시 반영

---

## 우선순위

```
Phase 1 (View)
  └── 디스플레이 화면 먼저 완성
       │
Phase 2 (Upload) ← Phase 1과 병행 가능
  └── 입력 화면 + CRUD
       │
Phase 3 (연동)
  └── 데이터 흐름 통합 + cleanup
```

## 핵심 기술 결정

| 항목 | 결정 |
|------|------|
| 데이터 저장 | `tb_home.hm_other` JSON |
| API | 기존 `home.php` PUT (hm_other 필드) |
| 7일 cleanup | 저장/조회 시 자동 필터링 |
| View 갱신 | 30초 시간, 1분 데이터 |
| 지난 회의 | opacity 0.3 흐리게 |
| 폰트 크기 | 상태 80px, 팀명 50px, 시간 40px |
